# Projectoolbox — Fitness Review & Improvement Plan (2026-07-10)

Method: full code/architecture audit + hands-on visual walkthrough as the user
(local dev against the production database, DTI project), benchmarked against
the Pilot reference implementation built 2026-07-09/10.

## Verdict

**Fit for purpose as a document-centric AI PM assistant with strong governance
— NOT yet fit for purpose as an end-to-end autonomous project engine.** The
selling surface (funnel, billing, approvals with context, reporting, EVM cost
mechanics, meeting intelligence, zero-fabrication guardrails) is genuinely
good and live. The core PM engine underneath (scheduling truth, real-world
work tracking, deterministic state) has structural gaps that show up on the
one live project today.

### What is strong (keep, market)
- Approvals with reasoning chain, impact scores, affected items, alternatives
- Reporting: grounded data pack, 8 audience-targeted types, schedules, export
- Cost mechanics: ESTIMATE/ACTUAL/FORECAST/COMMITMENT entries, honest EVM
  (EAC/ETC/VAC/TCPI; never fabricates AC), rate-card labour estimating
- DoD/DoR enforcement on task completion and sprint pull (422 gates)
- Living stakeholder register with sentiment time-series
- Zero-fabrication stack (validators, sanitisers, TBC discipline)
- Meeting processor with certainty-rated extraction and apply cards
- Sign-up → Stripe live billing (verified end-to-end 2026-07-08)

### What fails or is missing (found live)
1. **Schedule page on the live project (DTI) is EMPTY** — 0/0 tasks, "No
   phases yet" — while the PM tracker simultaneously reports 8 delivery
   tasks at 1%. The Gantt populates only from a manually synced schedule
   ARTEFACT, not from live tasks. The scheduling view and execution reality
   are two disconnected worlds.
2. **No real CPM.** `Task.isCriticalPath` and `dependencies` are written by
   the LLM when it emits a schedule CSV and parsed verbatim. No
   forward/backward pass, no float, no network diagram, no dependency-driven
   Gantt. "Critical path" in the UI is a model opinion, not a computation.
3. **No probabilistic forecasting** (no Monte Carlo; scenarios are fixed
   ×0.95/×1.15 multipliers).
4. **No field-work model.** Tasks aren't classified AI-executable vs
   real-world; no work instructions, no silent-task chases, no
   paste-progress intake with verbatim evidence quotes (meeting processor is
   the closest, without line-level evidence).
5. **Change control is heuristic.** CR flow exists and applies changes, but
   impact scores are LLM-supplied 1-4s (`schedule: hasDateChanges ? 3 : 1`),
   not computed Δdays/Δcost.
6. **Quality gaps**: no per-task acceptance criteria verification, no defect
   /snag log (qa-testing page is a re-skinned Issue list).
7. **Stale surfaces**: dashboard shows "waiting on N answers" cards from
   ARCHIVED projects/agents (Atlas 21, Decom 31, 34h+ old); 73 unread
   notifications; EVM page shows skeletons indefinitely for methodologies
   where EVM is disabled (no empty state).
8. **Dead surface area**: MS Project/SAP/ADO/Salesforce/ServiceNow
   integrations are "coming soon" stubs; blog/careers/webinars/docs/api/
   community all ComingSoon; SAFe + Kanban disabled; no comms-plan
   generator; no team-roster name injection.

### Architecture — does it work?
It works **because reconciliation keeps rescuing it**, which is the problem.
- State is dual-encoded: free-text `phaseStatus` + authoritative Phase
  timestamps; transitions are inferred by overlapping resolvers
  (next-action vs gate-guard) that must be manually kept in sync.
- A per-minute cron runs self-heal SQL (orphan relinks, stale-phase
  downgrades, premature-gate cancels, deployment unsticks) — the system
  spends every tick repairing drift rather than preventing it.
- The Vercel↔VPS seam fails silent: the VPS stub can mark jobs COMPLETED
  with no output; model failures fall to fallbacks that still burn credits
  (the 3-week billed-while-404 incident is this seam).
- The LLM owns load-bearing logic (critical path, dependencies, change
  impact) — auditable engines require code to own transitions and math,
  with the LLM writing content only (the Pilot pattern, proven this week).

No further analysis needed — the fault lines are identified. It needs the
migration below, not another study.

## The Plan

Pilot is the reference implementation: its core libraries are pure
TypeScript and port directly (`cpm.ts`, `analytics.ts` [EVM+Monte Carlo],
`progress.ts`, the register/impact/check-in patterns, approval-context API).

### P0 — Stabilise what users see (≈1 week)
1. Dashboard: exclude archived/closed projects' questions and prompts.
2. Schedule page: render from LIVE tasks (phase-window fallback like the
   tasks/gantt API already does), with artefact sync as enrichment — never
   an empty Gantt on a project with tasks.
3. EVM page: methodology-aware empty state (no infinite skeletons).
4. Notification hygiene: auto-read on project archive; digest grouping.
5. Billing integrity: no credit deduction on failed model calls/fallback
   paths; VPS job completion requires evidence (artefact/task ids) or the
   job is FAILED, not COMPLETED.

### P1 — Port the Pilot engine capabilities (2–4 weeks)
6. **Real CPM**: port `cpm.ts`; compute dependencies→ES/EF/LS/LF/float/
   critical from live tasks; LLM plans propose dependencies, code computes
   the path; replace `isCriticalPath` writes with computed values; add the
   network diagram; dependency-driven Gantt with Today/Target/Forecast.
7. **Monte Carlo** finish forecasting (port; seeded, P50/P85, hit-target %)
   on top of MetricsSnapshot trends.
8. **Field-work loop**: executor classification on tasks; work-instruction
   briefings; check-in chases on silence; paste-progress intake with
   verbatim evidence quotes + accept-per-line + buried-signals scan;
   blocked→risk escalation on the computed critical path.
9. **Computed change control**: keep the CR/approval flow, replace heuristic
   scores with computed Δcost (cost lines) and Δdays (CPM with/without).
10. **Quality**: per-task acceptance criteria + advisory verification on
    completion; real Defect model behind qa-testing; quality position on
    gate approvals.
11. **Smaller ports**: comms-plan generator from the live register; team
    roster (role→real name) injected into generation; FX booking with
    original-amount audit; approval expandable context for the types that
    lack it.

### P2 — Architecture hardening (3–6 weeks, behind feature flag)
12. Single-source state machine: retire free-text `phaseStatus`; Phase
    timestamps + one transition module (merge next-action resolver and
    gate-guard); every transition code-owned and logged.
13. Convert self-heal sweeps into ALERTS (drift telemetry) instead of
    silent fixes; burn down the causes until sweeps find nothing.
14. Harden the VPS seam: idempotent jobs, output-evidence contracts,
    circuit-breaker metrics surfaced to admin.

### P3 — Surface cleanup (parallel, low risk)
15. Remove or finish ComingSoon marketing pages (they erode trust).
16. Integrations: hide stubs behind a "request access" capture rather than
    fake tiles.
17. Re-enable Kanban (+SAFe) once P1 lands (Pilot's WIP-limit/flow-metrics
    pattern makes Kanban real); add PRINCE2 & AgilePM as first-class
    methodologies (templates exist in Pilot).
18. DTI (live project): unblock Sprint Zero — the user's own artefact
    reviews are the gate; surface that more loudly than a ticker.

### Sequencing note
P0 ships alone (small, user-visible). P1 items 6–7 unlock 8–10 (impact
analysis needs CPM). P2 runs behind a flag with the old resolver as
fallback until parity is proven on live projects.
